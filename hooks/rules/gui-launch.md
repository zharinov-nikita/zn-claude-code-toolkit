# Запуск GUI-приложений — на столе Claude

**Работает автоматический вотчер** (`desktop-watcher.ps1`, стартует SessionStart-хуком): любое новое окно, открытое процессами-потомками этой сессии, само переносится на виртуальный стол, где живёт окно-хост сессии (Zed/терминал). Обычные запуски (`cargo run`, dev-сборки, `Start-Process`) специальных действий не требуют.

Вотчер ловит и не-потомков (store-приложения, Windows Terminal, уже запущенный браузер): окно, появившееся в течение ~10 секунд после команды сессии, переносится тоже (кроме системных: explorer, поиск, Пуск, Taskmgr). Ручные запуски пользователя защищены двумя проверками: (1) если за ~2.5 секунды до появления окна был ввод с клавиатуры/мыши — окно пользовательское; (2) если родитель (или дед) процесса окна — explorer.exe (запуск ярлыком/Пуском/панелью задач, включая холодный старт браузера) — тоже пользовательское. Такие окна не трогаются. Обратная сторона: если пользователь активно печатает в момент, когда Claude открывает не-потомочное окно (терминал), оно может остаться на столе пользователя — тогда предложи перенести его вручную через VirtualDesktop.exe.

Правила:

- **Браузер открывай ТОЛЬКО новым окном** (`--new-window <url>` у Chrome/Edge), никогда вкладкой — вкладка откроется в существующем окне пользователя на его столе, а такое окно двигать нельзя.
- **Окна-диалоги** (класс `#32770`, например charmap) утилита виртуальных столов не адресует — перенести нельзя; предупреди пользователя.
- Обёртка `run-on-my-desktop.ps1 -FilePath "<exe>" -ProjectHint "<папка проекта>"` осталась как запасной способ для точечного запуска.

Утилита управления столами: `VirtualDesktop.exe` (`/List`, `/GetDesktopFromWindowHandle:<hwnd>`, `/GetDesktop:<n> /MoveWindowHandle:<hwnd>`) — для ручного переноса уже открытых окон. Скрипты и утилита лежат в каталоге, указанном строкой «Desktop scripts directory» ниже (`run-on-my-desktop.ps1`, `bin/VirtualDesktop.exe`).
